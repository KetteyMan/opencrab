# Design Critique

Diagnose what is working, what feels off, and what to change first. This skill is for critique and direction-setting, not immediate image generation.

## When to use
- Review a screenshot, mockup, webpage, or in-product UI
- Explain why a design feels cluttered, cheap, generic, weak, or confusing
- Prioritize visual changes before implementation
- Give PMs, designers, or engineers a crisp aesthetic read on the current state

## Workflow
1. Identify the screen's job: what the user is supposed to notice, understand, and do.
2. State the strongest signal first. Name one thing that already works.
3. Diagnose the biggest issues using `references/heuristics.md`.
4. Separate structural problems from purely decorative ones.
5. Prioritize fixes in order. Give concrete, implementable adjustments instead of vague adjectives.
6. If useful, propose a clearer visual direction in one sentence: mood, density, typography, and accent strategy.

## Critique rules
- Be direct but not theatrical.
- Do not list ten equal-weight issues. Prioritize the top few.
- Explain *why* something feels off, not only *that* it feels off.
- When the interface is close, recommend precise refinements. When it is fundamentally weak, say so clearly.
- If the product already has an established style, critique relative to that style rather than inventing a brand-new one.

## Output shape
- Overall read: one concise paragraph
- Top issues: 3 to 5 high-signal findings
- Recommended next moves: concrete edits, ordered by leverage

## Reference map
- `references/heuristics.md`: practical heuristics for hierarchy, layout, interaction clarity, and perceived quality.
